## Ideas / Observations

1. **Playbook gap — same as Scout:** `notion-api-direct.md` instructs "Use WebFetch to POST" but WebFetch cannot POST. This guide needs a working alternative. Suggestion: Add a `?data=BASE64` query parameter GET handler to the notion-proxy route so WebFetch (GET) can trigger it. This is a small code change.

2. **Alternative POST mechanism:** The Next.js proxy at `/app/api/notion-proxy/route.ts` could add a GET handler that accepts the JSON payload as a URL-encoded query parameter (`?payload=...`) and decodes it server-side. This would make WebFetch-based access work without requiring egress allowlist changes.

3. **Resilience recommendation:** Agent output files (like the scout's JSON files) are a good fallback pattern. Consider adding a similar "pending queue" pattern for follow-up agents — if a follow-up email is drafted but cannot be sent, save it to a local file for retry on the next session.

If any lead in "sent" status was emailed on or before 2026-03-26, they've now been waiting 10+ days for a follow-up. This is an active revenue risk.
